# Sonic Perpetual Trading UI - 实施路线图

## 📋 任务解读

这个任务要求构建一个能够处理高频实时数据的交易界面，重点考察：
- **实时数据处理能力**（WebSocket 高频更新）
- **性能优化**（避免重渲染、保持响应性）
- **架构设计**（状态管理、组件设计）
- **Web3 领域知识**（理解交易数据流）

## 🗺️ 实施路线图

### 第一阶段：项目初始化与基础架构
- 创建 Next.js/React 项目
- 配置 TypeScript
- 设置状态管理方案（Zustand/Jotai/Redux）
- 搭建基础 UI 框架

### 第二阶段：WebSocket 连接与数据层
- 实现 WebSocket 客户端
- 处理连接生命周期（连接/断开/重连）
- 实现序列号校验和快照重置逻辑
- 设计数据流架构（订单簿/交易数据）

### 第三阶段：核心 UI 组件开发
- 订单簿组件（Bids & Asks）
- 交易记录组件（Trade Tape）
- 订单输入面板（Mock Submit）
- 市场选择器
- 连接状态指示器
- 消息速率显示器

### 第四阶段：性能优化（至少实现 2 项）
- 虚拟化列表渲染（react-window/react-virtuoso）
- 批量状态更新（requestAnimationFrame/批处理）
- 颗粒化状态订阅（避免全局重渲染）
- Memoized 行渲染（React.memo/useMemo）
- WebSocket 重连处理

### 第五阶段：部署与文档
- GitHub 仓库准备
- Vercel 部署配置
- 编写详细 README

### 第六阶段：测试与优化
- 高负载测试（20-50 updates/s）
- 性能瓶颈分析
- 用户体验优化

## ✅ 待办任务清单

### 1. 初始化项目并配置开发环境
**描述**: 创建 Next.js/React + TypeScript 项目，配置 ESLint、Prettier、Tailwind CSS 等开发工具。选择并配置状态管理方案（推荐 Zustand 或 Jotai，轻量且适合高频更新）。

### 2. 设计并实现 WebSocket 客户端
**描述**: 实现 WebSocket 连接管理类，包括：连接/断开/自动重连逻辑、心跳检测、错误处理、序列号追踪。需要处理连接状态变化并通知 UI 层。

### 3. 实现订单簿数据模型和更新逻辑
**描述**: 创建订单簿数据结构（bids/asks），实现增量更新逻辑（delta application）、序列号校验、快照重置机制。需要保证数据一致性和正确性。

### 4. 开发订单簿 UI 组件（含虚拟化）
**描述**: 使用虚拟化列表（react-window 或 react-virtuoso）渲染订单簿。显示价格、数量、累计量。使用 React.memo 优化行渲染。实现价格闪烁效果（可选）。

### 5. 开发交易记录（Trade Tape）组件
**描述**: 实现虚拟化的交易记录列表，显示时间、价格、数量、买卖方向。处理每秒 5-20 条交易更新，保持滚动流畅性。

### 6. 实现批量状态更新机制
**描述**: 使用 requestAnimationFrame 或时间批处理策略，将高频 WebSocket 消息批量更新到 UI。避免每条消息都触发重渲染。需要权衡延迟和性能。

### 7. 开发订单输入面板和市场选择器
**描述**: 创建订单输入表单（价格、数量、买/卖），实现 mock 提交逻辑。添加市场选择器（单市场即可）。确保表单输入无延迟。

### 8. 实现连接状态指示器和消息速率显示
**描述**: 显示 WebSocket 连接状态（已连接/断开/重连中）。实时显示消息接收速率（msgs/s）。添加视觉指示器（绿/红/黄灯）。

### 9. 实现颗粒化状态订阅
**描述**: 使用状态管理方案的选择器功能（Zustand selectors），让组件只订阅需要的数据切片。避免订单簿更新触发无关组件重渲染。

### 10. 实现序列号间隙检测和快照重置
**描述**: 监控 WebSocket 消息序列号，检测间隙。当发现丢失消息时，请求新的快照并重置订单簿状态。确保数据一致性。

### 11. 性能测试和瓶颈分析
**描述**: 使用 React DevTools Profiler 分析渲染性能。测试 20-50 updates/s 和 5-20 trades/s 场景。记录 FPS、渲染时间、内存使用。识别性能瓶颈。

### 12. 优化用户体验和错误处理
**描述**: 添加加载状态、错误状态、空状态 UI。处理网络错误、WebSocket 断开等异常情况。确保 UI 层次清晰、信息易读。

### 13. 编写详细的 README 文档
**描述**: 包含：架构说明（数据流、状态管理）、性能优化决策、已识别瓶颈、10× 负载扩展策略、技术权衡说明、运行指南。

### 14. 部署到 Vercel 并创建 GitHub 仓库
**描述**: 创建 GitHub 公开仓库，配置 Vercel 自动部署。确保构建成功，获取公开 URL。在 README 中添加仓库链接、部署 URL、后端 API/WS URL。

### 15. 最终测试和代码审查准备
**描述**: 完整测试所有功能，验证实时更新流畅性。代码整理和注释。准备回答架构问题。测试修改部分应用的能力（评审可能要求现场修改）。

## 🔑 核心技术要点

### 1. 必须实现的组件
- ✅ 订单簿（实时更新的 Bids/Asks）
- ✅ 交易记录流（Trade Tape）
- ✅ 订单输入表单（可以是 Mock）
- ✅ 市场选择器
- ✅ 连接状态指示器
- ✅ 消息速率显示

### 2. 必须实现至少 2 项性能优化
推荐实现：
1. **虚拟化列表**（react-window）- 处理大量订单数据
2. **批量状态更新**（requestAnimationFrame）- 减少重渲染
3. **颗粒化订阅**（Zustand selectors）- 避免不必要的组件更新
4. **Memoized 渲染**（React.memo）- 优化行组件

### 3. 技术栈推荐

```
前端框架: Next.js 14+ (App Router)
语言: TypeScript
状态管理: Zustand / Jotai（轻量、适合高频更新）
虚拟化: react-window / react-virtuoso
样式: Tailwind CSS
实时通信: native WebSocket API
部署: Vercel
```

### 4. 性能目标
- 处理 **20-50 次/秒** 订单簿更新
- 处理 **5-20 次/秒** 交易数据
- 保持 UI 无卡顿、无闪烁
- 输入无延迟

### 5. 关键难点

#### WebSocket 序列号管理
检测消息丢失并重置状态，确保数据一致性

#### 高频更新性能
避免每次更新都触发全量重渲染，使用批处理和虚拟化

#### 订单簿增量更新
正确应用 delta 更新逻辑，维护有序的价格层级

#### 重连处理
断线后自动重连并恢复状态，处理快照重新同步

## ⚠️ 注意事项

### 1. WebSocket 后端选择
需要自己找或搭建一个 WebSocket 后端，可选方案：
- **公开加密货币交易所 API**（如 Binance、OKX WebSocket）
- **自建 Mock 服务器**（使用 Node.js + ws 库）
- **现有的测试环境**（如果项目方提供）

### 2. 代码可维护性
评审时可能要求**现场修改代码**，确保：
- 组件职责单一，易于修改
- 数据层与 UI 层分离
- 使用 TypeScript 类型保证安全性
- 代码注释清晰

### 3. 展示重点
- **实时数据处理能力**：展示如何优雅处理高频数据流
- **性能优化思维**：展示对渲染优化的深入理解
- **工程能力**：代码结构、错误处理、测试完整性
- **Web3 理解**：体现对 DeFi/交易系统的领域知识

### 4. 常见陷阱
- ❌ 每条 WebSocket 消息都触发状态更新
- ❌ 订单簿全量重渲染而非增量更新
- ❌ 没有虚拟化导致 DOM 节点过多
- ❌ 全局状态导致无关组件重渲染
- ❌ WebSocket 断线后无重连机制
- ❌ 没有序列号校验，数据可能不一致
